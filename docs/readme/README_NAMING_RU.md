# Конвенция именования StateHelper

Этот файл фиксирует единый стиль именования, чтобы проект дальше не разъезжался по стилю от модуля к модулю.

## Базовые правила

- использовать `snake_case`
- публичные функции начинать с `sh_`
- имя должно отражать слой, домен и действие
- один файл должен отвечать за один понятный участок логики

## Формат функций

Базовый шаблон:

`sh_<layer>_<domain>_<action>(...)`

Примеры:

- `sh_core_storage_save_settings`
- `sh_core_update_check`
- `sh_ui_notif_show`
- `sh_feature_reminder_add`
- `sh_mech_police_ticket_open`
- `sh_cmd_ticket_handle`

## Префиксы слоев

- `bootstrap` - запуск, доставка, обновление
- `core` - инфраструктура
- `feature` - пользовательский раздел
- `mech` - игровая механика
- `faction` - логика фракции
- `ui` - визуальный слой
- `cmd` - команда и ее обработка

## Как называть файлы

Имя файла должно идти от домена, а не от истории правок.

Хорошо:

- `Arizona_bootstrap/manifest.lua`
- `core/storage.lua`
- `features/reminder/reminder_core.lua`
- `Arizona_mechanics/police/smart_ticket.lua`
- `Arizona_factions/police/commands.lua`

Плохо:

- `utils2.lua`
- `new_file.lua`
- `final_fix.lua`
- `tmp_final.lua`

## Как называть состояния

Для таблиц состояния лучше использовать суффикс `_state`.

Примеры:

- `app_state`
- `tracking_state`
- `smart_ticket_state`

## Таблицы модулей

Если файл возвращает API-таблицу, имя таблицы должно совпадать с доменом файла.

Примеры:

- `local storage = {}`
- `local notifications = {}`
- `local smart_ticket = {}`

## Верхний блок файла

Для связующих файлов проекта полезно оставлять короткий верхний блок с описанием.

Минимально полезно указать:

- кодировку файла
- путь файла
- назначение
- роль в архитектуре
- короткие правила для этого файла

## Чего лучше не делать

- не смешивать `feature` и `mechanic` без причины
- не класть faction-логику в `core`
- не использовать абстрактные имена там, где можно назвать домен прямо
- не создавать новые перегруженные файлы под видом модулей
